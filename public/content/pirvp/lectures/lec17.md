# Лекция №17. Веб-безопасность

## 1. Введение в веб-безопасность

Веб-безопасность — это набор практик и технологий, которые позволяют защитить веб-приложение и данные пользователей от атак. Без базовой безопасности даже простое приложение может стать источником утечки данных или быть использовано злоумышленниками для атак на других пользователей.

**Ключевые моменты:**  
- Каждое веб-приложение потенциально уязвимо, если не принимать меры защиты.  
- Атаки могут быть направлены на пользователей (например, кража сессий), на сервер (например, SQL Injection) или на данные в сети (например, MITM — Man-in-the-Middle).  
- Простые меры безопасности значительно снижают риски, особенно при работе с внешними API и вводом данных от пользователей.

### 1.1 Зачем нужна защита веб-приложений

Защита веб-приложений нужна для того, чтобы:  
- Гарантировать конфиденциальность пользовательских данных.  
- Обеспечить целостность данных и корректность работы приложения.  
- Предотвратить несанкционированный доступ и модификацию данных.  
- Поддерживать доверие пользователей и соответствовать законодательству (например, GDPR).

**Ключевые моменты:**  
- Без защиты пользователи могут потерять свои данные, пароли, финансовую информацию.  
- Уязвимости в приложении могут позволить злоумышленнику выполнять произвольный код на сервере или браузере пользователя.  
- Простые меры (HTTPS, проверка данных, заголовки безопасности) уже дают значительный эффект.

### 1.2 Основные типы атак: XSS, CSRF, SQL Injection

**Cross-Site Scripting (XSS)** — это атака, при которой злоумышленник внедряет вредоносный скрипт в веб-страницу, которую видят другие пользователи.  
**Цель XSS:** украсть данные сессии, подменить содержимое страницы, перенаправить пользователя на другой сайт.

**Cross-Site Request Forgery (CSRF)** — это атака, при которой пользователь невольно выполняет нежелательные действия на сайте, где он аутентифицирован.  
**Цель CSRF:** использовать авторизацию жертвы для изменения данных, выполнения транзакций или действий от имени пользователя без его ведома.

**SQL Injection (SQLi)** — атака, при которой злоумышленник внедряет вредоносные SQL-запросы через пользовательский ввод.  
**Цель SQLi:** получить доступ к базе данных, украсть, изменить или удалить данные.

**Ключевые моменты:**  
- XSS влияет на пользователей и их браузеры, CSRF использует доверие сервера к пользователю, SQLi влияет на данные и сервер.  
- Все три атаки возможны из-за недостаточной фильтрации данных или неправильной конфигурации веб-приложения.  
- Понимание этих атак помогает проектировать защиту с нуля и применять меры безопасности до того, как приложение попадёт в продакшн.

## 2. XSS (Cross-Site Scripting)

XSS (Cross-Site Scripting) — это уязвимость веб-приложений, позволяющая злоумышленникам внедрять и исполнять произвольный скрипт (чаще всего JavaScript) в контексте сайта или приложения. Код выполняется в браузере пользователя и может:  
- Похищать конфиденциальную информацию (куки, токены сессий, личные данные).  
- Проводить фишинговые атаки, подделывая формы и ссылки.  
- Заставлять пользователя выполнять незапланированные действия (например, перевод средств).  
- Распространять вредоносные программы через веб-код.

XSS — это не единичный сценарий атаки, а целый класс уязвимостей с разными формами проявления.

### 2.1 Основные типы XSS

Чаще всего различают три вида XSS:  
1. Reflected XSS (отражённая)  
2. Stored XSS (хранимая)  
3. DOM-based XSS (основанная на манипуляциях с DOM)

#### 2.1.1 Reflected XSS (отражённая)

**Суть атаки:** скрипт «отражается» сервером обратно в браузер пользователя через специально сформированную ссылку или запрос. Злоумышленник отправляет ссылку с вредоносным кодом, который сервер возвращает «как есть», без фильтрации.

**Пример:**  
Форма поиска на сайте возвращает введённый пользователем запрос без экранирования:  
`http://example.com/search?q=<script>alert('XSS')</script>`  
Браузер исполнит код `alert`, что демонстрирует возможность внедрения произвольного JavaScript.

**Методы обнаружения:**  
- Проверка параметров URL и POST/GET запросов.  
- Анализ кода на предмет вывода данных без экранирования.  
- Использование сканеров безопасности (OWASP ZAP, Acunetix).

**Защита:**  
- Экранирование данных перед выводом в HTML.  
- Валидация пользовательского ввода на сервере.  
- Использование современных фреймворков с автоматической защитой (React, Angular, Vue.js).  
- Применение токенов и nonce для критичных операций.

#### 2.1.2 Stored XSS (хранимая)

**Суть атаки:** вредоносный скрипт сохраняется на сервере или в базе данных и выполняется у всех пользователей, которые просматривают соответствующий контент (например, комментарии, форумы, личные сообщения).

**Пример:**  
Злоумышленник оставляет комментарий:  
`<script>alert("Этот скрипт запускается у всех читателей!")</script>`  
Если сервер не фильтрует тег `<script>`, код выполняется у всех посетителей.

**Методы защиты:**  
- Фильтрация вводимых данных (блокировка опасных тегов и атрибутов).  
- Экранирование данных при выводе.  
- Использование Content Security Policy (CSP).  
- Ограничение прав пользователей для предотвращения выполнения скрипта.

#### 2.1.3 DOM-based XSS

**Суть атаки:** уязвимость возникает на стороне клиента при работе с DOM. Скрипт использует данные из `document.location`, `document.cookie` или `document.referrer` и вставляет их в HTML без экранирования.

**Пример:**
```html
<script>
  var content = window.location.hash.substring(1);
  document.getElementById("display").innerHTML = "Вы ввели: " + content;
</script>
```
Пользователь переходит по ссылке:  
`http://example.com/page.html#<script>alert('DOM-XSS')</script>`  
Код выполняется напрямую в браузере.

**Защита:**  
- Не использовать `innerHTML` для необработанных данных, применять `textContent`.  
- Избегать `eval()`, `document.write()` и других опасных конструкций.  
- Фильтровать и проверять данные из URL и cookies.  
- Использовать безопасные методы фреймворков (React, Angular, Vue.js).

### 2.2 Заголовки безопасности

Современные браузеры поддерживают HTTP-заголовки, которые помогают защитить веб-приложение от XSS и других атак:

1. **Content Security Policy (CSP)** — разрешает выполнение скриптов и загрузку ресурсов только из доверенных источников.  
2. **X-XSS-Protection** — встроенная фильтрация XSS (устаревший, но полезен для старых браузеров).  
3. **X-Frame-Options** — защита от clickjacking (`DENY`, `SAMEORIGIN`, `ALLOW-FROM`).  
4. **Strict-Transport-Security (HSTS)** — принуждает использование HTTPS.  
5. **Referrer-Policy** — управление передачей данных о реферере.  
6. **X-Content-Type-Options** — предотвращает MIME-снифинг (`nosniff`).  
7. **Permissions-Policy** — ограничивает доступ к браузерным API (камера, микрофон, геолокация).

## 3. CSRF (Cross-Site Request Forgery)

CSRF — это атака, при которой злоумышленник вынуждает пользователя выполнить нежелательное действие на веб-сайте, где он уже авторизован. Сервер доверяет запросам пользователя на основе его сессии или cookie, и именно этим злоумышленник пользуется.

**Пример:** пользователь авторизован в интернет-банке. Если он случайно откроет страницу злоумышленника, та может отправить запрос на перевод денег, используя его активную сессию. Сервер воспримет запрос как легитимный.

CSRF опасен тем, что злоумышленник не получает данных пользователя напрямую, но может выполнять любые действия от его имени, если они не требуют дополнительной проверки.

### 3.1 Механизм атаки

Атака CSRF работает через доверие сервера к браузеру пользователя. Рассмотрим последовательность:  
1. Пользователь входит на сайт `example.com` и его браузер получает сессионные cookie.  
2. Пользователь открывает сайт злоумышленника, например `evil.com`.  
3. На `evil.com` скрыт код (форма, изображение или скрипт), который отправляет запрос на `example.com`.  
4. Браузер автоматически добавляет cookie к запросу. Сервер `example.com` видит валидную сессию и выполняет действие, считая его легитимным.

**Ключевой момент:** CSRF срабатывает только при активной сессии пользователя. Без неё сервер отклоняет запрос.

**Примеры CSRF:**  
- Перевод денег на чужой счёт через скрытую HTML-форму или тег `<img>` с URL, вызывающим POST-запрос.  
- Смена пароля или email через автоматическую отправку POST-запроса с помощью JavaScript.  
- Подтверждение заказа или удаление аккаунта без ведома пользователя.

Эти примеры показывают, что атака может быть как заметной (смена пароля), так и незаметной (перевод средств), но в любом случае последствия зависят от доверия сервера к сессии пользователя.

### 3.2 Методы защиты

Существует несколько способов предотвращения CSRF:

- **CSRF-токены.** Для каждой формы или сеанса сервер генерирует уникальный токен. Он включается в форму и проверяется при получении запроса. Если токен отсутствует или не совпадает, запрос отклоняется.  
- **Проверка заголовков Origin и Referer.** Сервер анализирует, откуда пришёл запрос. Если он с другого домена, сервер блокирует его. Метод эффективен для POST-запросов, особенно критических.  
- **Флаг SameSite для cookie.** При установке `SameSite=Strict` браузер не отправляет cookie на сторонние сайты, предотвращая автоматическое выполнение запросов с чужих страниц.  
- **Разделение операций по типу запросов.** GET-запросы не должны изменять состояние сервера. Все действия, изменяющие данные, должны использовать POST, PUT или DELETE.  
- **Дополнительные подтверждения.** Для критических операций полезно требовать повторный ввод пароля или одноразовый код. Это создаёт дополнительный барьер для злоумышленника.

**Практические рекомендации:**  
- Всегда внедрять CSRF-токены на формы, которые изменяют данные.  
- Проверять Origin и Referer у всех POST-запросов.  
- Настраивать cookie с флагом SameSite.  
- Использовать двухфакторную аутентификацию для особо критичных действий.  
- Проводить тестирование с помощью инструментов безопасности, таких как Burp Suite или OWASP ZAP, чтобы выявлять возможные уязвимости.

## 4. SQL-инъекция

SQL-инъекция (SQL Injection, SQLi) — это одна из самых распространённых и опасных уязвимостей веб-приложений. Она возникает, когда приложение некорректно обрабатывает пользовательский ввод в SQL-запросах, позволяя злоумышленникам вмешиваться в работу базы данных.

**Цели SQL-инъекций:**  
- Просмотр конфиденциальной информации (данные пользователей, учетные записи администратора).  
- Изменение или удаление данных.  
- Обход авторизации и выполнение команд на сервере базы данных.

SQL-инъекции становятся возможными из-за:  
- Плохих практик программирования.  
- Отсутствия фильтрации или проверки пользовательского ввода.  
- Широкого распространения уязвимых приложений.

### 4.1 Механизм SQL-инъекции

SQL-инъекция работает через изменение структуры SQL-запроса за счёт пользовательского ввода.

**Пример запроса для авторизации:**  
```sql
SELECT * FROM users WHERE username = 'input_username' AND password = 'input_password';
```

Если пользователь введёт:  
**Username:** `'' OR '1'='1'`  
**Password:** `'' OR '1'='1'`

Запрос превратится в:  
```sql
SELECT * FROM users WHERE username = '' OR '1'='1' AND password = '' OR '1'='1';
```

Такой запрос всегда возвращает true, что позволяет злоумышленнику войти без правильных данных.

### 4.2 Практические примеры SQL-инъекций

1. **Обход аутентификации:**  

```sql
SELECT * FROM users WHERE username = '' OR '1'='1' AND password = '' OR '1'='1';
```
- Результат: всегда true.  
- Злоумышленник получает несанкционированный доступ.

2. **Извлечение данных из базы**  

```sql
SELECT * FROM users WHERE username = 'input_username';
```
Если ввести:  
`'' UNION SELECT null, username, password FROM users--`  
Запрос превращается в:  
```sql
SELECT * FROM users WHERE username = '' UNION SELECT null, username, password FROM users--;
```
- Результат: возвращаются все логины и пароли из таблицы `users`.

3. **«Слепая» SQL-инъекция**  

- Используется, когда приложение не возвращает ошибки, но его поведение зависит от выполнения запроса.  
- Злоумышленник постепенно извлекает данные, проверяя условия через поведение приложения (например, изменения в ответе или задержку).

### 4.3 Стратегии защиты

1. **Параметризованные запросы (Prepared Statements).** 

- SQL-код и данные разделяются, предотвращая изменение структуры запроса.  
- Пример на Node.js/Express с PostgreSQL:
```javascript
     app.post('/login', async (req, res) => {
       const { username, password } = req.body;
       const query = 'SELECT * FROM users WHERE username = $1 AND password = $2';
       const values = [username, password];

       try {
         const result = await pool.query(query, values);
         if (result.rows.length) {
           res.send('Logged in!');
         } else {
           res.send('Invalid credentials');
         }
       } catch (err) {
         console.error(err);
         res.status(500).send('Server error');
       }
     });
```
Пользовательский ввод безопасно подставляется в запрос, SQL-инъекция невозможна.

2. **Хранимые процедуры.**  

- Процедуры для взаимодействия с базой данных безопасны при корректной обработке входных данных.

3. **Проверка и очистка пользовательского ввода.**  

- Валидируйте формат данных (emails, числа и т.д.).  
- Экранируйте специальные символы, которые могут использоваться в SQL.

4. **Принцип наименьших привилегий.**  

- Учётная запись базы данных, используемая приложением, должна иметь минимальные права, чтобы ограничить последствия возможной SQL-инъекции.

5. **Регулярные проверки безопасности.**  

- Проводите аудит кода и тестирование на наличие SQLi-уязвимостей.

## 5. CORS (Cross-Origin Resource Sharing)

CORS — это механизм веб-браузеров, который управляет доступом к ресурсам на сервере с других доменов (кросс-доменные запросы).  
По умолчанию браузеры блокируют запросы с одного домена на другой из соображений безопасности (Same-Origin Policy).  
CORS позволяет серверу явно разрешать такие запросы.

**Пример:**  
- Сервер: `api.example.com`  
- Фронтенд: `app.example.com`  
- Без CORS браузер заблокирует запрос, если сервер не разрешит доступ.

### 5.2 Основные заголовки CORS

1. **Access-Control-Allow-Origin**  

- Определяет, с каких источников разрешены запросы.  
- Пример: `Access-Control-Allow-Origin: https://app.example.com`

2. **Access-Control-Allow-Methods**  

- Указывает, какие HTTP-методы разрешены.  
- Пример: `Access-Control-Allow-Methods: GET, POST, PUT`

3. **Дополнительные заголовки:** 

- `Access-Control-Allow-Headers` — какие заголовки можно отправлять.  
- `Access-Control-Allow-Credentials` — разрешает отправку куки и авторизационных данных.

### 5.3 Настройка CORS в Express.js

Для удобства в Node.js/Express используется пакет `cors`.

**Пример базовой настройки:**

```javascript
const express = require('express');
const cors = require('cors');
const app = express();

// Разрешить все источники
app.use(cors());

// Разрешить только определённый фронтенд
// app.use(cors({ origin: 'https://app.example.com', methods: ['GET','POST'] }));

app.get('/data', (req, res) => {
  res.json({ message: 'CORS работает!' });
});

app.listen(3000, () => console.log('Server running on port 3000'));
```

## 6. HTTPS и SSL/TLS

### 6.1 Отличие HTTP и HTTPS

**HTTP (HyperText Transfer Protocol)** — протокол передачи данных между клиентом и сервером, который не предусматривает шифрование. Это значит, что информация, передаваемая через HTTP, может быть перехвачена злоумышленниками или изменена по пути.

**HTTPS (HyperText Transfer Protocol Secure)** — защищённая версия HTTP, которая использует шифрование через протоколы SSL или TLS. Благодаря этому обеспечивается безопасная передача данных, а также проверка подлинности сервера.

**Основные преимущества HTTPS:**  
- Конфиденциальность данных — передаваемая информация шифруется, что исключает возможность её перехвата.  
- Целостность данных — любые изменения данных в процессе передачи становятся заметными.  
- Аутентификация сервера — клиент проверяет подлинность сервера с помощью сертификата, чтобы убедиться, что соединение происходит с доверенным ресурсом.

### 6.2 Как работает SSL/TLS

SSL (Secure Sockets Layer) и TLS (Transport Layer Security) — это протоколы, которые обеспечивают безопасное соединение между клиентом и сервером. Механизм работы включает несколько этапов:

1. **Handshake (рукопожатие)** — клиент и сервер договариваются о способах шифрования и обмениваются ключами.  
2. **Аутентификация сервера** — сервер отправляет цифровой сертификат, который проверяется через доверенные центры сертификации (CA).  
3. **Сессионный ключ** — после проверки сертификата клиент и сервер создают общий ключ для симметричного шифрования всех последующих данных.  
4. **Шифрование данных** — весь обмен информацией проходит в зашифрованном виде, что исключает возможность перехвата или подмены.

Таким образом, SSL/TLS обеспечивает защищённый канал связи, позволяя безопасно передавать конфиденциальную информацию между клиентом и сервером.

### 6.3 Сертификаты и настройка сервера

Для работы HTTPS сервер должен иметь SSL/TLS-сертификат — цифровой документ, подтверждающий подлинность сервера и содержащий открытый ключ для шифрования.

**Существуют разные типы сертификатов:**  
- **Self-signed (самоподписанные)** — создаются самим сервером, подходят только для тестирования и локальной разработки.  
- **CA-signed (от центра сертификации)** — выданы доверенным центром сертификации, используются на продакшн-сайтах.  
- **Let’s Encrypt** — бесплатные сертификаты с автоматическим обновлением, широко применяются для веб-сайтов.

**Основные задачи сертификата:**  
- Подтверждение подлинности сервера.  
- Установка защищённого канала связи между клиентом и сервером.  
- Обеспечение доверия пользователей и безопасности передаваемых данных.
